## 1. 模型回复变成 `[object Promise]`

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/components/ChatMessageBubble.vue`
**哪个功能**：聊天气泡里渲染 markdown

**现象**：助手回复气泡里显示一行 `[object Promise]`，正文一字没有。

**为什么**：
我们用了 `marked` 这个库把 markdown 文本（比如 `**加粗**`）转成 HTML。
`marked` 从 v15 开始，**默认变成异步**：调用 `marked.parse(text)` 返回的不是字符串，而是一个"承诺"对象（Promise）。
我们把这个 Promise 直接塞进 `v-html`，浏览器只会调用它的 `.toString()`，得到 `[object Promise]`。

**怎么修**：
强制让它走同步：
```ts
marked.use({ async: false });
const rawHtml = marked.parse(text, { async: false }) as string;
```
两处都得加：一是 `marked.use`（全局开关），二是每次调用时再补一次 `async: false`，双保险。

**给老太太的领悟**：升级第三方库要看 changelog（更新说明），尤其是大版本号变了，默认行为常常变。

## 2. SSE 流式回复在 EventSource 上没收到 token

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/composables/useChatStream.ts` + 后端 `@/Users/leila/Documents/coding/tongtong/backend/app/api/routes/chat.py`
**哪个功能**：聊天流式输出

**现象**：浏览器发请求，后端日志显示在吐 token，但前端气泡是空的。

**为什么**：
浏览器原生 `EventSource` API 只支持 **GET** 方法。
我们一开始只写了 `POST /chat/stream`，前端 `EventSource('/chat/stream', {body: ...})` 根本没法传 body。

**怎么修**：
后端**同时**保留 GET 和 POST 两个端点：
- POST 留给以后改用 `fetch + ReadableStream` 的实现
- GET 走 `?session_id=...&query=...` 参数，浏览器 EventSource 直接连

**给老太太的领悟**：选 API 协议前先确认浏览器能不能用。EventSource 看着方便，但只能 GET。

## 3. 推到 messages 数组里的对象，token 来了 UI 不刷新

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/pages/WorkspacePage.vue`
**哪个功能**：流式追加助手消息

**现象**：写法
```ts
const message = { role: 'assistant', content: '' };
messages.value.push(message);
streamHandle = openStream(..., {
  onToken: (t) => { message.content += t; }, // ← 看着对，但 UI 不动
});
```
气泡始终空白，刷新后能看到完整回答（说明数据写进了，但响应式没触发）。

**为什么**：
`messages.value.push(message)` 之后，**Vue 把数组里那个对象包了一层 Proxy**（响应式代理）。
我们手里的 `message` 还是**原始对象**，改它不会通知 Vue 更新。

**怎么修**：
push 之后，**立刻从数组里把那个 Proxy 版本取出来**，对它做修改：
```ts
messages.value.push(raw);
const reactive = messages.value[messages.value.length - 1];
// 后面 reactive.content += token 就会触发更新
```

**给老太太的领悟**：Vue 3 的响应式建立在 Proxy 上。push 进 ref 数组之后，**返回值在数组里、不在你手里那个变量里**。

## 4. `<think>...</think>` 流式过程中渲染断断续续

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/components/ChatMessageBubble.vue`
**哪个功能**：把模型思考过程做成可折叠块

**现象**：
模型先吐 `<think>用户问的是...`，下一帧才补 `</think>正文`。
中间这一帧用户看到的是一个**没闭合的 HTML 标签**，浏览器把后面所有内容吞进 think 节点里。

**为什么**：
正则 `/<think>([\s\S]*?)<\/think>/g` 只能匹配**已闭合**的 think。流式过程中没闭合那一刻，正则匹配不到，原样穿过去到 marked，HTML 就坏了。

**怎么修**：
两段正则，二选一兜底：
```ts
// 先尝试匹配闭合的
result = raw.replace(/<think>([\s\S]*?)<\/think>/g, ...);
// 兜底：还没闭合的当作"思考中…"折叠块先显示
result = result.replace(/<think>([\s\S]*)$/, ...);
```

**给老太太的领悟**：处理流式数据要考虑"半截状态"，正则只匹配完整数据是不够的。

## 5. 切换会话后，再发消息标题被覆盖

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/backend/app/services/history_service.py`
**哪个功能**：会话标题持久化

**现象**：用户刚把会话改名为"差旅报销咨询"，发了下一条消息，标题立刻被改成"那住宿呢？"。

**为什么**：
`append_turn` 方法每次都会执行 `session["title"] = title`，title 来自前端传的 query 前 24 字。
也就是说**每条消息都会用最新 query 覆盖标题**。

**怎么修**：
只在"会话第一次创建"时设标题：
```python
is_new_session = session_id not in user_sessions
session = user_sessions.setdefault(session_id, {...})
if is_new_session:
    session["title"] = title
```

**给老太太的领悟**：副作用要看清楚是"创建时执行"还是"每次都执行"。setdefault 容易给人"只创建一次"的错觉，但后面的赋值是每次都跑的。

## 6. 侧边栏滚出视口顶部"消失"了

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/pages/WorkspacePage.vue`
**哪个功能**：左侧导航栏布局

**现象**：右侧聊天内容多了，整页一滚，侧边栏顶部的 logo / 导航就被卷出屏幕，只看到底部"退出登录"卡片浮在中间。

**为什么**：
我用了 `<a-layout-sider>`，它本身**不会自动 sticky**。整个 layout 是普通的 flex row，sider 高度跟内容一起被拉长，所以 sider 顶部会跟着页面滚走。

**怎么修**：
给 sider 加上：
```css
.workspace-sider {
  position: sticky !important;
  top: 0;
  align-self: flex-start;  /* 关键：不要被 stretch 拉满 */
  height: 100vh !important;
}
```
`align-self: flex-start` 是关键——不写它，flex 子项默认 stretch 到父容器一样高，sticky 就失效。

**给老太太的领悟**：sticky 要工作，得满足两个条件：① 自身是 flex/普通子项不被拉伸；② 所有祖先节点都不能有 `overflow: hidden`。

## 7. 对话区无限增长，整页变得超长

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/style.css` + `@/Users/leila/Documents/coding/tongtong/frontend/src/pages/WorkspacePage.vue`
**哪个功能**：聊天消息滚动

**现象**：消息一多，对话区直接顶到下面去，输入框被挤到屏幕外。

**为什么**：
我换 antd 时把原来 Tailwind 的 `overflow-y-auto pr-2 space-y-4` 工具类摘掉了，
换成了 class `chat-scroll-area`，但 `.chat-scroll-area` **只设了 max-height，没设 overflow-y: auto**。
浏览器默认 `overflow: visible`，max-height 形同虚设——内容超了就溢出来。

**怎么修**：
```css
.chat-scroll-area {
  max-height: calc(100vh - 460px);
  overflow-y: auto;       /* ← 这条是核心 */
  overflow-x: hidden;     /* 顺手防止超长代码横向撑开 */
  display: flex;
  flex-direction: column;
  gap: 16px;              /* 替代之前的 space-y-4 */
}
```

**给老太太的领悟**：`max-height` 单独无效，必须配 `overflow: auto/hidden/scroll` 才生效。这是 CSS 最常踩的坑之一。

## 8. antd 主题色配上去后，按钮还是默认蓝

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/App.vue`
**哪个功能**：全局主题

**现象**：在 `App.vue` 里设了 `colorPrimary: '#b30000'`，但 `<a-button type="primary">` 还是 antd 默认蓝。

**为什么**：
我一开始没用 `<a-config-provider>` 包整个应用，而是只在 main.ts 的 createApp 里 `.use(Antd)`。
antd 的主题 token 必须通过 `<a-config-provider :theme="...">` 注入到组件树里。

**怎么修**：
```vue
<a-config-provider :theme="antdTheme" :locale="zhCN">
  <RouterView />
</a-config-provider>
```
**包住整个应用**，否则只对子节点生效。

**给老太太的领悟**：很多组件库的全局配置都是"放在最外层 Provider"模式，看官方文档第一段一般会写。

## 9. `<a-layout-sider>` 默认有自己的暗黑底色

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/pages/WorkspacePage.vue`
**哪个功能**：侧边栏外壳

**现象**：自己给 AppSidebar 设了白底，但 sider 周围还是有一圈深色 `#001529`。

**为什么**：
`<a-layout-sider>` 默认背景就是 `#001529`（"夜空蓝"），用来配 antd 黑色管理后台主题。
我们的 AppSidebar 比 sider 窄一点，露出来一条深色"裙边"。

**怎么修**：
```css
.workspace-sider {
  background: transparent !important;
}
.workspace-sider :deep(.ant-layout-sider-children) {
  display: flex;
  height: 100%;
}
```
`:deep()` 是 Vue scoped CSS 穿透到子组件内部 class 的写法（旧版叫 `>>>` 或 `::v-deep`）。

**给老太太的领悟**：组件库的默认样式很多是"管理后台范儿"，做产品时几乎都要覆盖一遍。

## 10. SSE 用 GET 时鉴权 token 没法塞 header

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/composables/useChatStream.ts`
**哪个功能**：流式聊天的鉴权

**现象**：登录后调对话，后端 401。

**为什么**：
浏览器原生 `EventSource` 不支持自定义 header（Authorization: Bearer xxx 没法加）。
普通 fetch 我们都通过 `Authorization` header 走 token，但 EventSource 没这个 API。

**怎么修**：
两条路：
- A：改用 `fetch + ReadableStream`，能自定义 header
- B：保持 EventSource，把 token 当 query 参数传：`/chat/stream?token=xxx`，后端从 query 取
当前项目走的是 B（开发环境）。生产环境建议升级 A，token 在 query 里有泄漏风险（日志、Referer）。

**给老太太的领悟**：浏览器原生 API 不一定满足生产环境需求。EventSource 简单但有局限。

## 11. Vite 大改依赖后，HMR 显示"connection lost"

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/package.json` 改动后
**哪个功能**：开发服务器热更新

**现象**：装完 antd 后，浏览器控制台一直红字 `[vite] server connection lost`。

**为什么**：
Vite 启动时会缓存依赖图（`node_modules/.vite`）。装新包后图变了，老的 HMR socket 找不到原来的 module 引用。

**怎么修**：
1. 终端 Ctrl+C 停掉 vite
2. 重新跑 `npm run dev`
3. 浏览器 **Cmd+Shift+R 硬刷新**（绕过缓存）

如果还不行，删掉 `frontend/node_modules/.vite` 缓存目录再启动。

**给老太太的领悟**：装新依赖后**重启 dev 服务器 + 硬刷新浏览器** 是基本操作。

## 12. DOMPurify 把 `<details>` 标签当不安全过滤掉

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/frontend/src/components/ChatMessageBubble.vue`
**哪个功能**：思考过程折叠块

**现象**：
我们把 `<think>...</think>` 替换成 `<details><summary>思考过程</summary>...</details>`，
经过 marked 后再过 DOMPurify，结果 `<details>` 和 `<summary>` 全没了，只剩纯文字。

**为什么**：
DOMPurify 默认有自己的安全白名单。`<details>` 在某些版本里被认为是"不常见标签"被剥离。

**怎么修**：
显式声明它们是允许的：
```ts
DOMPurify.sanitize(html, {
  ADD_TAGS: ['details', 'summary'],
  ADD_ATTR: ['open'],
});
```

**给老太太的领悟**：安全过滤库很容易"宁错杀不放过"，遇到合法 HTML 没渲染出来，先看是不是被它清理掉了。

## 13. 后端意图分类闲聊也走业务，回答僵硬

**在哪个文件**：`@/Users/leila/Documents/coding/tongtong/backend/app/chains/intent_chain.py` + `@/Users/leila/Documents/coding/tongtong/backend/app/graphs/conversation_graph.py`
**哪个功能**：意图识别

**现象**：用户说"你好"，助手回复一段"很抱歉我没找到相关信息"。

**为什么**：
意图分类把 chitchat（闲聊）当成"信息不足"处理，硬塞进 RAG 检索流程，自然查不到，就出"信息不足"模板回答。

**怎么修**：
在 chain 里给 chitchat **专门一条出口**，直接进 LLM 自由生成，不走 RAG。
关键词触发的业务（薪酬/知识库/差旅/个人）保留分支不变。

**给老太太的领悟**：意图分类**别用一个分支管所有事**，闲聊和业务要分开走。

## 总结：通用规律

1. **每升级一个第三方库都要看 changelog**（marked 改默认异步、antd 改组件 API 都坑过我）
2. **响应式系统的 Proxy 包装**——push 之后要从数组里拿回来才是响应式
3. **CSS 的 max-height/overflow 是一对**，单独写 max-height 90% 的概率不工作
4. **sticky 需要 flex-start + 没有溢出祖先**——记住这两条规则少走半天弯路
5. **流式数据要考虑半截状态**——前端正则、后端 chunking 都是
6. **组件库默认样式 ≠ 产品样式**，覆盖是常态，做好 :deep() 心理准备
7. **安全过滤要白名单确认**——DOMPurify、CSP、CORS 都同理
