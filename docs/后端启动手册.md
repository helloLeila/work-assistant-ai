# 企业智能办公助手后端启动手册

这份手册只讲一件事：**怎么把后端单独跑起来**。

如果你暂时不想管前端，只想确认接口、登录、聊天流、知识库和权限链路能不能工作，就照这份手册做。

## 1. 后端是什么

后端程序入口在 [backend/app/main.py](/Users/leila/Documents/coding/tongtong/backend/app/main.py)。

启动后，它会提供这些接口：

- `/api/health`：健康检查
- `/api/overview`：系统概览
- `/api/auth/login`：登录
- `/api/chat/stream`：流式聊天
- `/api/chat/history`：会话历史
- `/api/knowledge/upload`：上传知识库文件
- `/api/knowledge/list`：查看知识库文件
- `/api/knowledge/{doc_id}`：删除知识库文件

## 2. 先知道两种启动方式

你可以二选一：

### 方式 A：最小启动

适合你现在这种情况：

- 想先把后端跑起来
- 没有真实商旅接口
- 只有 MiniMax Key
- 甚至没有 Docker

这个模式下，系统会自动使用这些回退能力：

- PostgreSQL 不可用：退回本地 SQLite
- Milvus 或 Embedding 不可用：退回本地词法检索
- 商旅接口没填：退回本地订单确认

### 方式 B：完整启动

适合你已经装好 Docker，想把 PostgreSQL、Redis、Milvus 也一起启动。

## 5.1 为什么要用 Docker

先说结论：

**你不是必须用 Docker 才能把后端跑起来。**

你之所以会看到 Docker 命令，是因为它能帮你一键启动这些依赖服务：

- PostgreSQL
- Redis
- Milvus
- etcd
- minio

### 什么时候建议用 Docker

当你想跑“更完整”的环境时，用 Docker 很方便。原因是：

1. PostgreSQL 不用你手动安装
2. Redis 不用你手动安装
3. Milvus 这种向量数据库本地手装更麻烦
4. 团队里每个人用同一套环境，最省事

### 什么时候可以不用 Docker

当你只是想先把后端 API 跑起来、确认登录和聊天接口可用时，可以**先不用 Docker**。

因为这个项目已经做了回退策略：

- PostgreSQL 不可用：自动退回 SQLite
- Milvus 不可用：自动退回本地词法检索
- 商旅接口没填：自动退回本地订单确认

所以对你现在来说，最稳的做法其实是：

1. 先不跑 Docker
2. 先把后端用 `uvicorn` 跑起来
3. 确认 `/api/health`、登录、聊天流都正常
4. 之后再决定要不要补 Docker 环境

## 12. 如果后端启动失败，优先排查什么

按这个顺序查：

1. 是否执行过一次 `pip install -r requirements.txt`（首次必装）
2. `backend/.env` 文件有没有复制出来
3. `OPENAI_BASE_URL` 和 `OPENAI_MODEL` 有没有乱填
4. 端口 `8000` 有没有被别的程序占用
5. 启动是否卡在 `Waiting for application startup.`

### 12.1 `ERROR: [Errno 48] Address already in use`

含义：8000 端口被旧的 uvicorn 进程占着了。常见于 `--reload` 模式下父进程被 Ctrl+C，但 worker 子进程没一起退出。

直接一把梭：

```bash
make kill-backend && make dev-backend
```

`make kill-backend` 会同时清理：

- 占用 8000 端口的所有 PID（不分 LISTEN / CLOSED 状态）
- 名字匹配 `uvicorn app.main:app` 的遗留进程

如果想手动看一下占用是谁：

```bash
lsof -nP -iTCP:8000
```

### 12.2 启动卡在 `Waiting for application startup.`

含义：lifespan 阶段被外部依赖阻塞住了。最常见原因是 **Embedding 接口不可用**：

- MiniMax 国内版（`api.minimaxi.com`）在 OpenAI 兼容接口下**没有 Embedding 模型**
- 但 `backend/.env` 里仍然填了 `OPENAI_EMBEDDING_MODEL=text-embedding-3-small` 之类

解决办法：把 `backend/.env` 里的 `OPENAI_EMBEDDING_MODEL` **留空**，知识库会自动回退到本地词法检索。改完必须重启后端才会生效（`get_settings()` 有缓存）。

### 12.3 模型一调就 404

OpenAI 兼容接口的 `OPENAI_MODEL` **大小写敏感**：

- 正确：`MiniMax-M2.7`
- 错误：`minimax-m2.7`、`MINIMAX-M2.7`

确认对应供应商的官方模型名再填。

### 12.4 改了 `backend/.env` 但行为没变

后端没有重启。`Ctrl+C` 停掉旧进程后重新 `make dev-backend`。

### 12.5 Bocha 联网搜索每次都 TimeoutError

**症状**：问"天气"等需要实时信息的问题，前端思考块里飞过：

```
正在调用 Bocha 搜索「天气」
联网搜索失败（TimeoutError），改用本地知识
```

而且每次都是 1.5 秒左右秒挂。

**根因**：`BOCHA_TIMEOUT_SECONDS` 最初被设成 `1.5`，与意图分类（utility 模型）的 1.5s 阈值对齐。但这是认知错位：

- 意图分类是**本地小模型链路**，1.5s 是合理上限；
- Bocha 联网搜索是**出网 HTTP 调用**，要走 DNS + TLS + 远端检索，实测 RTT 通常 2–6 秒。

1.5s 的超时几乎在请求还在握手阶段就被掐断，所以"每次必挂"。

**方案**：把超时改到 8 秒（给慢路径留余量），同时保留"超时即降级"的兜底策略，不让用户卡死。

涉及两处：
- [backend/.env](/Users/leila/Documents/coding/tongtong/backend/.env)：`BOCHA_TIMEOUT_SECONDS=8.0`
- [backend/app/core/config.py](/Users/leila/Documents/coding/tongtong/backend/app/core/config.py) 的 `bocha_timeout_seconds` 默认值同步改为 `8.0`

**教训**：给外部 HTTP 调用设超时时，先用 `backend/test_bocha.py` 这类脚本测一次真实 RTT，再设阈值。不要凭"和别的链路对齐"的直觉来定。
